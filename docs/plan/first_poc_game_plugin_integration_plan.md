# 首个 PoC 游戏插件接入方案

## 1. 方案目标

本文档用于把首个 PoC 游戏插件接入方案定成一套可直接执行的实现蓝图，目标是围绕 **Minecraft Java 版** 跑通以下闭环：

1. 游戏内发起绑定
2. Web 端完成账户确认
3. 插件与主站稳定通信
4. 游戏事件上报并触发金币结算
5. 商城兑换后由插件执行发奖
6. 后台封禁可联动到游戏服

PoC 阶段重点不是把所有玩法都做全，而是先把 **账户、事件、钱包、商城、运营控制** 这几条主链路打通，并沉淀可复用的插件接入协议。

---

## 2. 为什么选 Minecraft 作为首个 PoC

结合现有规划文档，首个 PoC 选择 Minecraft Java 版最合适，原因如下：

- 插件生态成熟，开发资料与社区实践丰富
- 玩家身份稳定，适合基于 UUID 建模
- 可快速验证 `/bind`、上线/下线、在线时长奖励、发奖命令等核心流程
- 本地与测试服调试成本低，适合 MVP 迭代
- 后续扩展到 Rust、FiveM 时，可复用主站接口、签名与事件协议设计

本阶段暂不覆盖 Bedrock，也不做跨版本复杂兼容，优先保证 Java 版 PoC 可跑通。

---

## 3. PoC 范围边界

### 3.1 本期必须交付

- Minecraft 插件基础框架
- 插件鉴权与服务端注册
- 游戏内绑定命令与配对码流程
- 玩家上线/下线/在线时长等基础事件上报
- 主站事件接收与奖励结算接口联通
- 商城兑换后的插件发奖回执机制
- 后台封禁联动到游戏服
- 插件心跳/状态上报

### 3.2 本期明确不做

- 多游戏共用插件运行时
- 复杂反作弊与风控策略
- 大规模事件异步批处理优化
- 多服联邦同步
- Bedrock 兼容
- 丰富化经济规则与多类商城发奖

PoC 只验证一条最小闭环，后续再扩展复杂能力。

---

## 4. 目标游戏与运行环境

## 4.1 目标游戏

- 游戏：Minecraft Java Edition
- 服务端建议：Paper / Purpur（优先 Paper）
- 插件语言：Java / Kotlin 均可，PoC 优先 Java 以降低接入门槛
- 玩家身份主键：Minecraft UUID

## 4.2 运行前提

插件部署在社区自管 Minecraft 服务器中，通过 HTTPS 调用 GameMulti API，并使用分配给该服务器的签名凭证完成鉴权。

建议区分三层实体：

1. **Game Server**：一台实际游戏服实例
2. **Plugin Installation**：某服上的插件安装实例
3. **Player Identity**：Minecraft UUID 对应的平台游戏身份

这样后续扩展到 Rust、FiveM 时，平台侧模型更容易统一。

## 5. 接入点设计

## 5.1 插件侧接入点

Minecraft 插件至少需要实现以下接入点：

### 命令入口

- `/gm bind`
  - 玩家在游戏内发起绑定
  - 返回短期有效的配对码或确认链接
- `/gm profile`
  - 查看当前绑定状态
- `/gm wallet`（可选）
  - 查看主站同步来的金币摘要

### 生命周期事件

- `onEnable`
  - 读取配置、初始化 HTTP 客户端、注册命令/监听器、上报插件上线
- `onDisable`
  - 上报插件下线（最佳努力）

### 玩家事件

- 玩家加入服务器
- 玩家离开服务器
- 在线时长累计
- 可选扩展行为：击杀、任务完成、签到等

### 管理动作执行器

- 发奖命令执行器
- 封禁/解封执行器
- 配置热加载或状态刷新（可选）

---

## 5.2 主站侧接入点

主站需要提供一组面向插件接入的 API：

### 注册与状态

- `POST /api/game-servers/register`：注册或刷新服务器接入信息
- `POST /api/game-servers/heartbeat`：插件心跳与状态上报

### 绑定流程

- `POST /api/plugin/binding-sessions`：创建绑定会话
- `GET /api/plugin/binding-sessions/{token}`：查询绑定状态
- `POST /api/web/binding-sessions/{token}/confirm`：Web 端确认绑定

### 事件与奖励

- `POST /api/plugin/events`：上报玩家行为事件
- `POST /api/plugin/reward-deliveries/{id}/ack`：发奖结果回执

### 运营控制

- `GET /api/plugin/player-state/{uuid}`：查询封禁/权限/绑定摘要
- `POST /api/plugin/command-poll` 或 Webhook/长轮询方案：拉取待执行平台命令

PoC 阶段可以先采用 **插件主动拉取 + 主站入库记录** 的简化模式，降低双向实时通信复杂度。

---

## 6. 端到端事件流

## 6.1 账户绑定流

1. 玩家在 Minecraft 中执行 `/gm bind`
2. 插件调用主站创建绑定会话
3. 主站返回：
   - `binding_token`
   - `pair_code`
   - `expire_at`
   - 确认链接
4. 插件将配对码或链接展示给玩家
5. 玩家登录 GameMulti 主站，在 Web 页完成确认
6. 主站将平台账户与该 Minecraft UUID 绑定
7. 插件查询绑定状态，或在下一次轮询中拿到确认结果
8. 插件提示玩家“绑定成功”

### 最低要求

- 绑定 token 短时有效
- 玩家未登录 Web 时不能直接绑定
- 重复绑定要有幂等和冲突提示
- 解绑与改绑不纳入首期 PoC

---

## 6.2 玩家在线事件流

1. 玩家加入服务器
2. 插件记录 join 事件并上报
3. 插件本地累计在线时长
4. 达到一个结算窗口后，上报 duration 事件
5. 主站根据奖励规则计算金币
6. 钱包入账并生成流水
7. 用户中心展示最新金币变化

### PoC 简化建议

- 先用固定规则：每在线 10 分钟奖励 X 金币
- 由服务端做幂等，避免重复上报导致重复发放
- 插件只负责采集和发送，不内置复杂奖励规则

---

## 6.3 商城兑换发奖流

1. 玩家在主站商城下单并消耗金币
2. 主站生成待发奖任务
3. 插件轮询待执行命令/奖励
4. 插件在游戏服执行对应命令，例如发放物品、称号、权限组
5. 插件回执执行结果
6. 主站更新订单/发奖状态

### PoC 最小商品类型

建议只做一类最容易验证的商品：

- 游戏内命令发放型商品
  - 例如：给玩家执行一条控制台命令
  - 示例：发基础礼包、称号或白名单权限

这样能快速跑通订单到插件执行的最短路径。

---

## 6.4 封禁联动流

1. 后台对平台账户或绑定身份执行封禁
2. 主站生成待下发的封禁动作
3. 插件轮询获取平台命令
4. 插件执行服务器内封禁/踢出
5. 插件回执封禁执行结果
6. 主站记录审计日志

PoC 阶段重点是证明“后台控制可以影响游戏服”，不必一开始覆盖复杂封禁维度。

---

## 7. 账户、金币、商城对接边界

## 7.1 账户边界

插件只感知以下账户信息：

- 玩家 Minecraft UUID
- 玩家当前昵称（仅作展示）
- 是否已绑定
- 绑定到哪个平台账户（可只返回脱敏摘要）
- 当前是否可执行平台动作（例如是否被封禁）

插件 **不直接处理**：

- Web 登录态
- 邀请码注册逻辑
- 平台账户密码与凭证
- 论坛 SSO 逻辑

这些全部留在主站处理。

---

## 7.2 金币边界

插件负责：

- 采集与发送可结算事件
- 接收奖励回执结果（可选展示）

主站负责：

- 奖励规则配置
- 幂等判断
- 金币账本入账
- 钱包余额与流水查询
- 风控与补单

原则上插件不维护金币真账本，避免多端状态分叉。

---

## 7.3 商城边界

插件负责：

- 执行主站下发的“可在游戏内完成”的奖励命令
- 回传执行成功/失败结果

主站负责：

- 商品定义
- 库存与上下架
- 订单创建与扣费
- 发奖任务创建与重试
- 失败订单处理与审计

因此商城系统是主站能力，插件只是游戏侧履约执行器。

---

## 8. 插件与主站通信约束

## 8.1 鉴权建议

PoC 阶段建议采用以下组合：

- 每个 Game Server 分配独立 `server_id`
- 分配 `access_key` / `secret` 或签名密钥
- 请求头携带时间戳、随机串、签名
- 主站校验签名与重放窗口

这样比单纯写死 Bearer Token 更适合后续多服扩展与密钥轮换。

## 8.2 幂等与重试

插件上报行为事件时，建议带：

- `event_id`
- `server_id`
- `player_uuid`
- `occurred_at`
- `event_type`

主站按 `event_id` 或业务主键做幂等，允许插件网络失败后安全重试。

## 8.3 失败处理

- 插件请求超时：本地排队重试或降级记录日志
- 主站 4xx：视为请求参数或鉴权错误，进入人工排查
- 主站 5xx：可退避重试
- 长时间无法通信：插件状态标记为 degraded，并在管理台展示

---

## 9. 验收方式

本 PoC 接入方案验收建议分为五个最小场景：

### 场景 A：绑定流程

- 新玩家进入服内执行 `/gm bind`
- 主站生成配对码
- 玩家在 Web 完成确认
- 插件侧收到绑定成功结果

### 场景 B：在线奖励

- 已绑定玩家在线达到阈值
- 插件成功上报 duration 事件
- 主站成功入账金币
- 用户中心可看到钱包流水

### 场景 C：商城发奖

- 玩家使用金币下单一个命令型商品
- 主站创建发奖任务
- 插件成功执行命令并回执
- 订单状态变为已完成

### 场景 D：封禁联动

- 后台封禁某玩家
- 插件轮询拿到封禁指令
- 游戏服执行踢出/封禁
- 主站审计日志记录完整

### 场景 E：心跳与状态

- 插件启动后完成注册与心跳
- 主站能看到该游戏服在线
- 插件下线后状态可识别为离线或超时

### 验收结论标准

通过上述五个场景，即可认定首个 PoC 游戏插件接入路线成立，后续可以进入实现和扩展阶段。

---

## 10. 下一步实现拆分建议

建议把 PoC 实现拆成以下几个可并行的开发任务：

### 任务 1：平台插件接入协议与 API 定稿

交付物：

- 插件接入协议文档
- 鉴权签名说明
- 事件 DTO / 错误码
- 心跳、绑定、事件、发奖回执接口定义

### 任务 2：主站后端插件接入模块

交付物：

- game server 注册与密钥管理
- 绑定会话 API
- 事件接收与幂等处理
- 发奖任务队列与回执接口
- 封禁动作下发接口

### 任务 3：Minecraft 插件 PoC

交付物：

- `/gm bind` 命令
- join/quit/duration 事件监听
- API 客户端
- 命令轮询与发奖执行
- 基础配置文件与日志

### 任务 4：Web 端绑定确认与用户中心展示

交付物：

- 绑定确认页
- 绑定成功/失败状态页
- 用户中心游戏绑定信息卡片
- 钱包流水展示联动

### 任务 5：商城与后台最小联调

交付物：

- 最小命令型商品
- 订单 -> 发奖任务链路
- 后台封禁操作到插件命令下发
- 审计日志页面或日志核验方案

### 任务 6：测试与演示环境

交付物：

- 本地或 dev 环境 Docker 化基础设施
- 一套 Paper 测试服
- 演示脚本与验收 checklist

---

## 11. 方案结论

首个 PoC 游戏插件接入建议定为 **Minecraft Java 版**，并采用“主站为中心、插件负责采集与执行”的模式推进。

PoC 阶段的关键不是把业务做大，而是先验证：

- 绑定能跑通
- 事件能上报
- 奖励能入账
- 商城能履约
- 后台控制能落到游戏服

只要这五条主链路成立，GameMulti 就具备了从规划走向工程落地的最小平台基础。后续扩展 Rust、FiveM 或更多游戏时，复用主站账户、钱包、商城和运营体系即可。
