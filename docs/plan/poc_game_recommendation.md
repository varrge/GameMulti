# 首个 PoC 游戏接入建议

## 1. 目标

本文件用于回答一个关键问题：

> 首个应该选哪个游戏做接入 PoC，为什么？

PoC 阶段的目标不是一开始覆盖所有游戏，而是：

1. 跑通游戏内发起绑定 -> Web 确认
2. 跑通事件上报 -> 金币结算 -> 钱包入账
3. 跑通商城兑换 -> 插件发奖
4. 跑通后台封禁 -> 游戏服执行

因此，首个 PoC 游戏应优先选择：
- 插件能力成熟
- 玩家身份可稳定获取
- 服务器生态清晰
- 行为事件容易采集
- 调试成本较低

---

## 2. 推荐顺序

### 第一推荐：Minecraft

#### 原因
1. 插件生态成熟，服务端开发资料丰富
2. 玩家身份体系明确，Java 版可围绕 UUID 建模
3. 适合快速实现：
   - `/register`
   - `/bind`
   - `/pair`
   - 上线/下线事件
   - 在线时长统计
   - 简单奖励命令
4. 开发与调试体验相对友好
5. 适合先跑通“绑定 + 时长奖励 + 发奖 + 封禁”完整链路

#### 风险点
- 若未来重点是 Steam 系游戏，则首个 PoC 在 Steam 身份体系上的复用度略低
- 需要提前明确 Java 版 / Bedrock 版差异，建议首期只做 Java 版 UUID 模型

---

### 第二推荐：Rust

#### 原因
1. 更贴近 Steam 系游戏模型
2. 若未来重点围绕 SteamID64，则与主站身份和平台扩展关系更直接
3. 行为事件与服务器状态对接价值高

#### 风险点
- 插件开发与调试门槛通常比 Minecraft 更高
- 首期可能更适合作为第二个 PoC，而不是第一个

---

### 第三推荐：FiveM

#### 原因
1. 适合验证多身份系统与自定义事件模型
2. 社区玩法和运营扩展性强

#### 风险点
- 身份模型可能更复杂
- 自定义玩法多，首期产品边界更容易膨胀

---

## 3. 建议结论

### 最推荐方案
- **首个 PoC：Minecraft（Java）**
- **第二阶段扩展：Rust**

### 理由
这样可以：
1. 用 Minecraft 快速跑通插件接入与绑定闭环
2. 用 Rust 再补 Steam 系身份与行为模型
3. 避免一开始就陷入多游戏复杂接入造成的拖延

---

## 4. 首个 PoC 必须验证的能力

不管选哪个游戏，PoC 必须至少验证以下能力：

1. 游戏内命令发起绑定
2. token / 配对码创建
3. Web 端确认绑定
4. 玩家上线/下线事件上报
5. 基础在线时长奖励
6. 钱包入账
7. 商城兑换一类游戏内物品
8. 后台封禁 -> 游戏服执行
