# GameMulti 项目基线与 Roadmap

## 1. 文档目标

本文档用于把 GameMulti 当前阶段的项目基线和推进路线定稿，作为后续任务拆分、仓库初始化、并行开发和阶段验收的统一参考。

这里的“项目基线”包括：

- 当前项目定位与阶段判断
- MVP 范围与非目标
- 核心模块边界
- 模块依赖关系
- 阶段划分与里程碑
- 各阶段优先级与建议进入条件

当前判断：GameMulti 仍处于 **规划定稿 -> MVP 落地准备** 阶段，下一步应从“文档齐备”切换到“按主链路拆模块开工”。

---

## 2. 项目基线

## 2.1 项目定位

GameMulti 是一个面向多游戏联机社区的统一平台，目标不是做单一游戏官网，而是围绕以下能力建立平台化基础：

1. 邀请制社区准入
2. 主站统一账户体系
3. 游戏内发起绑定、Web 端确认
4. 游戏行为事件采集与金币结算
5. 商城兑换与游戏内履约
6. 论坛联动与后台运营控制
7. 多游戏插件扩展能力

也就是说，GameMulti 的核心不是单一页面，而是 **主站 + 插件 + 钱包 + 商城 + 运营后台 + 社区论坛** 组成的统一系统。

---

## 2.2 当前阶段判断

结合现有文档产出，当前项目已经完成了以下准备：

- 项目总体规划
- 技术选型方向
- 核心数据库模型草案
- API 与模块边界草案
- 阶段性里程碑拆分
- 首个 PoC 游戏建议
- 协作与部署基线建议

因此当前阶段不再是“要不要做、怎么想”的探索期，而是“把 MVP 主链路拆出来并正式进入实现”的阶段。

### 当前基线结论

- **产品阶段**：MVP 准备阶段
- **工程阶段**：规划已基本可用，适合初始化代码仓库与开发环境
- **首个 PoC 游戏**：Minecraft Java 版
- **首要目标**：优先跑通邀请制账户、绑定、事件、金币、商城最小闭环

## 2.3 MVP 范围

MVP 建议只覆盖最小可验证业务链路，不追求一步到位。

### MVP 必须覆盖

- 邀请码注册与登录
- 游戏账户绑定（Minecraft PoC）
- 绑定确认页与用户中心绑定展示
- 玩家行为事件上报
- 在线时长类金币奖励
- 钱包余额与流水
- 商城下单与命令型商品发奖
- 后台最小运营动作（邀请码管理、封禁联动、日志）
- 基础部署、监控、日志、回滚能力

### MVP 明确不做

- 多游戏同时上线
- 复杂反作弊风控
- 高阶营销系统
- 多层级商城玩法
- 大规模微服务拆分
- 复杂推荐、社交、消息中心
- 过早的高可用集群化改造

原则是先把“能跑通的最小平台”做好，再扩展玩法和规模。

## 3. 核心模块划分

建议把系统分成六个核心模块。

## 3.1 账户与身份模块

职责：

- 邀请制注册
- 登录与会话
- 平台账户管理
- 游戏身份绑定关系
- 封禁状态管理

这是所有业务链路的入口模块，也是其他模块的依赖基础。

---

## 3.2 游戏插件接入模块

职责：

- 游戏服注册与鉴权
- 绑定会话创建
- 玩家事件上报
- 心跳与状态上报
- 平台命令下发
- 发奖回执

首个落地对象为 Minecraft Java 插件，后续扩展到 Rust / FiveM 等。

---

## 3.3 钱包与账本模块

职责：

- 金币余额管理
- 账本流水
- 事件结算入账
- 扣费与退款记录
- 幂等控制与审计

钱包模块必须尽量保持“平台单一真账本”，不要把账务状态下放到插件侧。

---

## 3.4 商城与订单模块

职责：

- 商品管理
- 订单创建
- 金币扣费
- 发奖任务
- 发奖状态回执
- 失败重试与人工处理

MVP 只需要支持最小命令型商品，不做复杂商品类型。

---

## 3.5 论坛与社区联动模块

职责：

- Discourse SSO
- 用户同步
- 社区身份映射
- 封禁联动（可逐步接入）

该模块对长期社区运营重要，但不必阻塞 MVP 主链路优先落地。

---

## 3.6 后台运营与审计模块

职责：

- 邀请码管理
- 游戏服管理
- 商品管理
- 封禁/解封
- 审计日志
- 平台状态查看

该模块负责让系统可运营、可排错、可追责。

---

## 4. 模块依赖关系

下面是建议的依赖优先级：

1. **账户与身份模块**
   - 最先落地，是绑定、钱包、商城、论坛的前置依赖
2. **游戏插件接入模块**
   - 与账户模块联动，跑通绑定和事件入口
3. **钱包与账本模块**
   - 承接事件奖励与订单扣费
4. **商城与订单模块**
   - 依赖钱包，形成消费闭环
5. **后台运营与审计模块**
   - 依赖账户、插件、钱包、商城数据
6. **论坛与社区联动模块**
   - 可在 MVP 后半段或阶段 4 再重点推进

### 依赖结论

如果要排开发顺序，建议严格按：

**账户 -> 绑定/插件 -> 钱包 -> 商城 -> 后台 -> 论坛**

这样最符合当前业务闭环目标。

## 5. Roadmap 总览

## 阶段 0：规划与工程基线定稿

### 目标

把文档、技术方向、协作方式、仓库与环境基线全部定下来，为正式开发扫清歧义。

### 关键内容

- 项目规划文档定稿
- 数据模型与接口边界确认
- 首个 PoC 游戏确认
- 协作规范与部署基线确认
- 后续代码仓库初始化方案确认

### 完成判定

- 核心规划文档齐备
- 各角色可以按文档继续拆开发任务
- 不再因大方向反复摇摆而阻塞开工

### 当前状态

- **基本完成**

---

## 阶段 1：邀请制账户与绑定 MVP

### 目标

跑通从邀请注册到游戏账户绑定的主链路。

### 范围

- 邀请码注册
- 登录与基础会话
- 绑定会话创建
- Web 绑定确认页
- 用户中心绑定状态展示
- 最小管理接口

### 里程碑结果

玩家能够：

- 被邀请注册平台
- 登录主站
- 从游戏内发起绑定
- 在 Web 完成确认
- 在用户中心看到绑定结果

### 优先级

- **P0，必须最先落地**

### 进入下一阶段条件

- 绑定闭环稳定可演示
- 绑定状态可查询、可审计
- 基础异常场景可处理（超时、重复绑定、未登录）

---

## 阶段 2：首个 PoC 游戏插件接入

### 目标

接通 Minecraft Java 插件，实现主站与游戏服稳定通信。

### 范围

- 插件鉴权
- `/gm bind` 命令
- join/quit/duration 事件上报
- 插件心跳
- 状态同步
- 平台命令下发基础能力

### 里程碑结果

主站能与首个 Minecraft 测试服完成绑定与事件联动。

### 优先级

- **P0，和阶段 1 强相关，可并行准备、串行验收**

### 进入下一阶段条件

- 插件在 dev/staging 环境稳定运行
- 事件上报可幂等处理
- 基础日志和排错手段可用

---

## 阶段 3：金币与商城闭环

### 目标

形成“玩游戏 -> 得金币 -> 花金币”的最小经济闭环。

### 范围

- 在线时长奖励规则
- 钱包余额与流水
- 商城商品展示
- 金币扣费下单
- 发奖任务与回执

### 里程碑结果

玩家能通过游戏行为获得金币，并在商城兑换至少一种游戏内奖励。

### 优先级

- **P0，MVP 成败关键阶段**

### 进入下一阶段条件

- 奖励入账正确
- 重复事件不会重复发奖
- 订单、扣费、发奖、回执链路闭合

---

## 阶段 4：论坛与后台运营闭环

### 目标

补齐社区与运营控制能力，让平台从“可演示”走向“可管理”。

### 范围

- Discourse SSO
- 用户同步
- 后台管理入口
- 邀请码管理
- 封禁联动
- 审计日志

### 里程碑结果

主站、论坛、游戏服和后台形成基本统一运营闭环。

### 优先级

- **P1，重要但不阻塞最早 MVP 演示**

### 进入下一阶段条件

- 管理动作可追踪
- 社区账号流转顺畅
- 封禁策略基本打通

---

## 阶段 5：多游戏扩展与上线准备

### 目标

在首个 PoC 跑通后，进入平台化扩展和上线前加固阶段。

### 范围

- 第二/第三游戏接入
- 规则扩展
- 稳定性与性能测试
- 监控告警补强
- 部署与回滚完善
- 上线准备清单

### 里程碑结果

项目从单游戏 PoC 演进为可扩展的多游戏平台，并具备正式上线准备条件。

### 优先级

- **P1/P2，视 MVP 成功情况推进**

---

## 6. 里程碑优先级结论

从当前到 MVP，建议优先级排序如下：

1. **P0：账户与绑定闭环**
2. **P0：Minecraft 插件接入闭环**
3. **P0：金币与商城闭环**
4. **P1：后台与审计能力**
5. **P1：论坛联动**
6. **P2：多游戏扩展、性能与风控深化**

这意味着近期所有任务拆分、仓库初始化和环境准备，都应围绕前三项展开，不要被边缘功能分散注意力。

## 8. 结论

GameMulti 当前已经完成了从“概念想法”到“可执行规划”的过渡，项目基线可以定为：

- 以统一主站为中心
- 以 Minecraft Java 版为首个 PoC 游戏
- 以账户绑定、事件奖励、商城履约为 MVP 核心闭环
- 以后台与论坛作为第二层运营能力补齐
- 以多游戏扩展作为 MVP 验证后的平台化阶段目标

接下来最重要的不是继续扩写大而全的规划，而是按这条 Roadmap 把仓库、接口、环境和开发任务正式落地。
