# 游戏联机社区项目总体规划

## 1. 项目目标

本项目旨在搭建一个完整的游戏联机社区 Web 平台，覆盖官网、用户中心、游戏账号绑定、社区金币体系、兑换中心、论坛系统打通、游戏服务器数据接入、后台管理系统，以及封禁联动与运营能力。

核心业务链路如下：

1. 玩家在游戏内通过插件命令发起注册或绑定。
2. 插件生成短链接或配对码，玩家跳转到主站完成账户确认。
3. 游戏插件持续上报玩家行为数据与服务器状态。
4. 后端根据规则将游戏行为结算为社区金币。
5. 玩家在主站商城或论坛体系内消费金币，兑换 VIP、论坛称号、游戏道具等。
6. 后台统一完成玩家管理、邀请码管理、封禁联动、流水审计与运营管理。

---

## 2. 项目核心原则

### 2.1 邀请制社区

本项目不是开放注册社区，而是邀请制社区。注册模型应围绕“谁邀请了谁、用了哪张邀请码、邀请码如何被追踪和审计”设计。

因此，系统设计中应将邀请码体系作为用户准入的一等公民，而不是附属校验项。

### 2.2 主站统一身份中心

主站账户是整个平台的统一身份中心。论坛、游戏绑定、金币系统、兑换系统、后台管理等能力都围绕主站用户展开。

### 2.3 游戏身份按游戏特性绑定

不同游戏使用不同的稳定玩家标识：

- Steam 系游戏优先使用 SteamID64
- Minecraft 使用 UUID
- FiveM 使用 license 或稳定 identifier
- 其他游戏按各自可验证、可长期稳定的唯一 ID 绑定

不将所有游戏身份强行统一成 SteamID64，但允许 Steam 登录作为主站登录方式之一。

### 2.4 游戏内发起绑定为主流程

玩家注册或绑定以游戏内发起为主：

- 玩家在游戏聊天框输入注册/绑定命令
- 插件调用后端创建绑定会话
- 后端返回短链接或配对码
- 玩家在 Web 端登录/注册并确认绑定
- 完成主站账户与游戏身份的正式关系建立

### 2.5 绑定安全规则

已确认的绑定规则如下：

- 绑定会话有效期：5 分钟
- 链接 token 单次使用
- 配对码长度：6~8 位字母数字
- 配对码与链接统一按 5 分钟失效处理
- 解绑需管理员审核，或在冷却时间到后允许用户自行解绑

## 3. 平台模块划分

项目按以下五个一级模块推进：

### 3.1 主站官网与用户系统

包含：

- 官网首页
- 登录/注册
- 邀请码校验
- 用户中心
- 游戏绑定页面
- 金币余额与 VIP 展示

### 3.2 论坛系统

包含：

- 开源论坛选型与部署
- 与主站统一身份中心打通
- 登录状态同步
- 导航融合
- 勋章/称号/封禁状态同步

### 3.3 游戏接入层

包含：

- 游戏插件协议
- 绑定会话创建
- 玩家行为事件上报
- 服务器状态上报
- 发奖/封禁命令下发
- 多游戏适配器设计

### 3.4 金币与兑换系统

包含：

- 金币账本
- 奖励规则
- 自动结算
- 商城兑换
- 发货与补偿任务

### 3.5 运营后台

包含：

- 玩家搜索与详情
- 邀请码管理
- 金币流水查询
- 封禁/解封
- 服务器状态总览
- 审计日志

---

## 4. 首期实施范围建议

建议优先完成 MVP 闭环，而不是一开始铺满所有游戏和功能。

首期重点建议包括：

1. 完成主站基础框架与邀请制注册模型
2. 确认首个可落地的 PoC 游戏接入
3. 跑通游戏内发起绑定 -> Web 确认绑定
4. 跑通金币账本与基础奖励结算
5. 跑通商城兑换与一类奖励发放
6. 跑通论坛 SSO
7. 跑通后台玩家查询、邀请码管理、封禁联动基础版

---

## 5. 推荐实施策略

建议按照“先跑通一条完整闭环，再横向扩多游戏”的思路推进：

1. 先选一个最容易做插件与事件上报的游戏做 PoC
2. 先实现邀请制注册 + 绑定流程 + 金币账本 + 商城兑换 + 后台查询
3. 再扩展更多游戏适配器、更多奖励规则、更多商城品类
4. 最后补齐高级风控、复杂运营活动与多游戏联动能力
